# 投研数据库维护规范

## 维护链路

```
单篇速读 → 更新事件 → 指标卡 → 领域阶段摘要 → 主题判断 → 更新日志
```

链路说明：

| 层级 | 输入 | 输出 |
|------|------|------|
| 单篇速读 | 原始研报 PDF/网页 | 压缩版：数据、边际变化、图表证据 |
| 更新事件 | 单篇速读 | 单点事实变化，按时间排列 |
| 指标卡 | 同指标多篇速读+更新事件 | 同一变量连续跟踪，含历史值与当前值 |
| 领域阶段摘要 | 同领域多个指标卡 | 该领域整体判断（已验证/待验证/被证伪） |
| 主题判断 | 跨领域摘要 | 投资结论与行动建议 |
| 更新日志 | 所有变更 | 变更事实清单（机器可读） |

链路规则：
- 信息流动方向为单向抽象，不逆向回写
- 每层只消费下层输出，不跨层依赖
- 上层更新不自动覆盖下层内容

---

## 日期字段定义（必须严格区分）

| 字段 | 定义 | 适用场景 |
|------|------|----------|
| `effective_date` | 数据实际对应的统计日期或事件发生日 | 指标值、报告期、事件基准日 |
| `report_date` | 该文件首次写入系统的日期 | 研报速读、更新事件、主题判断的创建基准 |
| `updated_at` | 文件内容最后一次修改的时间戳 | frontmatter `updated` 字段，所有文件通用 |

三者不得混用。例：
- 3月制造业PMI（effective_date: 2026-03-31），4月1日发布研报（report_date: 2026-04-01），指标卡4月5日补充了新订单数据（updated_at: 2026-04-05）。

## 语义处理

所有语义处理默认交由 LLM 完成，不部署本地语义模块。

具体分工：

| 任务 | 执行者 |
|------|--------|
| 从文本中提取关键变量名、数值、变化量 | LLM |
| 多篇内容归纳为一句话判断 | LLM |
| 判断边际变化方向（增强/减弱/反转） | LLM |
| 判断阈值是否被突破 | LLM |

本地仅负责：
- 文件结构存储（路径、frontmatter 字段）
- YAML frontmatter 格式校验
- 双向链接有效性校验

## 什么时候更新什么

### 更新事件

- 有单点事实变化（数据发布、政策宣布、价格突破等）
- 不判断影响方向，只记录事实

### 更新指标卡

- 同一指标出现新值
- 指标方向强化或削弱了当前判断
- 阈值被打穿或失效条件被触发

### 更新领域阶段摘要

- 多个指标卡的变化累积出了新阶段判断
- 出现了跨指标的新逻辑链
- 原有核心矛盾被解决或被新矛盾取代

### 更新主题判断

- 跨领域的投资结论需要调整
- 形成了新的交易主题
- 原有主题被证伪或升级

### 不更新

- 报告重复已有观点，无新数据
- 报告情绪性评论，无关键变量
- OCR 明显残缺，无法确认关键数据

## 判断状态标记

统一使用三类判断：

| 状态 | 含义 |
|------|------|
| `已验证` | 至少两份报告或后续数据相互印证，方向一致 |
| `待验证` | 仅一份报告支持，或判断成立但缺后续数据 |
| `被证伪` | 后续数据明显反向，或关键阈值被打穿 |

---

## 回原文复核图表的条件

- OCR 后图表编号跳跃
- 标题缺失或单位不清
- 同一指标正文与图表冲突
- 关键数字只出现一次且影响判断大
- 准备据此做交易，但图表证据不够扎实

优先回看：给出阈值的、历史分位的、同比/环比拐点的、行业空间/估值/出货预测的。

---

## 维护硬规则

- 不直接覆盖原始速读里的证据链
- 不把作者观点直接当成事实数据
- 不把"市场涨了"直接写成"逻辑被验证了"
- 所有更新写清：当前值、对比值、变化方向、对投资的直接指导
