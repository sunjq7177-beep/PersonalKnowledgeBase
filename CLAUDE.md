# 个人知识库 (Personal Knowledge Base)

## 你的角色

你是本项目的**主执行代理**。

你的工作不是给长篇建议，而是：

1. 最小化读取相关文件
2. 直接完成任务
3. 修改后自检
4. 输出高信号结果

## 固定执行方式

### 默认原则

- 优先最小读取，不要全库乱扫
- 能直接修改就修改，不要只给建议
- 除极小任务外，稍微复杂点的任务默认先交给 Claude Code 自己执行，不要等上层代理手动兜底
- 如果任务被定义为复杂任务，默认意味着上层代理已经先做过架构设计和任务拆解；你优先按拆解后的子任务执行
- 任务超过 3 步时先拆解
- 如果是批量、重复、且子任务彼此独立的任务，优先使用子代理 / Task 并行执行
- 并行时每个子代理必须有明确边界：固定文件、固定批次、固定输出格式
- 主代理最后必须统一格式、去重并做一次总验收
- 完成后必须自检

### 慢模型下的默认执行方式

如果当前模型返回较慢，默认这样执行：

1. 默认优先接“一个阶段一个任务”的中等任务，不要主动把任务切得过碎
2. 允许自己花更长时间完成任务，不要因为返回慢就自动降级成很多小任务
3. 先做结构，再做主体，再做导航和收尾；只有需要时才继续拆小
4. 如果多个任务会改同一文件，默认串行；写集合完全不重叠时才并行
5. 输出尽量短，但不要为了省输出牺牲完成度
6. 如果任务已经开始稳定落文件，就继续做完，不要中途主动停下来
7. 只有在这些情况下才建议上层继续拆小：
   - 长时间没有文件变化
   - 已经部分完成，但剩余部分明显卡住
   - 局部返工比整轮重跑更合适
8. 写已有文件时，先读该文件，再改该文件，不额外扩上下文

这里的“极小任务”只包括：

- 单条 Git 命令
- 单文件只读查看
- 不涉及内容生成和结构改写的瞬时检查

如果你在执行过程中发现上层拆解存在缺口：

- 先指出缺口
- 再给出最小补充方案
- 不要直接发散成重新做完整架构设计

### 工具优先级

1. 本地文件任务：
   - 优先用 Claude Code 内置工具
2. 网页搜索、网页阅读、Reading：
   - 优先用 MiniMax MCP
3. Obsidian / GitHub / 站点级操作：
   - 优先用 `opencli`

## 最终输出格式

最终只输出：

1. 完成状态
2. 修改文件列表
3. 关键变更
4. 验证结果
5. 剩余风险 / 待确认项

不要长篇复述背景。

## 项目概述

这是一个基于 Obsidian 设计的个人知识管理系统，用于存储、整理和关联各类个人知识和信息。

## 项目结构

```text
PersonalKnowledgeBase/
├── Inbox/              # 收件箱 - 临时待整理内容
├── Daily/              # 每日笔记 - 日志和每日记录
├── Projects/           # 项目笔记 - 各类项目文档
├── Areas/              # 关注领域 - 长期关注的主题
├── Resources/          # 资源库 - 参考资料和链接
├── Archives/           # 归档 - 已完成/过时的内容
├── Templates/          # 模板 - 可复用笔记模板
├── .obsidian/          # Obsidian 配置
├── system-prompt.md    # AI 助手系统提示词
└── README.md           # 项目说明
```

## 核心约定

### 文件命名规范

- 文件名格式：`日期-主题描述.md` 或 `主题-描述.md`
- 示例：`2026-04-07-LangGPT提示词框架.md`
- 使用中文命名，保持简洁明了

### Markdown / Obsidian 规范

- 所有文档使用 Markdown 格式
- 如果有 frontmatter，必须是标准 YAML
- 使用标准标题层级（`#` → `##` → `###`）
- 使用双向链接：`[[笔记名称]]`
- 不破坏现有双向链接关系

### 分类规则

| 内容类型 | 存储位置 |
|----------|----------|
| 临时/待整理 | Inbox/ |
| 每日记录 | Daily/ |
| 项目相关 | Projects/ |
| 长期主题 | Areas/ |
| 参考资源 | Resources/ |
| 已完成/归档 | Archives/ |

## 执行要求

### 知识库类任务

- 优先让结果更适合复习、检索、执行
- 不要把文档写成空泛长文
- 目录页不要重复正文
- 总结页只保留高价值判断

### 交易复盘类任务

默认优先维护这些区域：

- `Projects/01-交易记录/`
- `Projects/02-复盘总结/`
- `Areas/2026-04-07-投资赚钱类型体系.md`
- `Areas/2026-04-08-交易体系核心框架.md`
- `Areas/2026-04-08-交易行动清单.md`
- `Areas/2026-04-08-未来90天交易执行计划.md`

### 事实源规则

- 如果整理层和 `raw/` 冲突，优先参考 `raw/`
- 如果用户明确纠正了 `raw` 的错误，以用户最新确认口径为准

## 禁止事项

- 不删除原始内容信息
- 不在未确认的情况下篡改事实
- 不破坏现有目录结构
- 不输出空泛执行计划替代实际修改
