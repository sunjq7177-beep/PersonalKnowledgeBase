# 个人知识库 (Personal Knowledge Base)

## 当前执行链路

本项目默认采用以下协作方式：

1. 用户只给需求。
2. 上层调度代理负责拆解任务。
3. `Claude Code CLI` 是主执行器，负责实际搜索、修改、整理和自检。
4. 上层调度代理负责验收、提炼结果、决定是否二次返工。

也就是说：

- **默认不要绕开 Claude Code 直接手改**
- **默认先调用 `cc-run`**
- **如果是批量且可拆分的重复任务，默认优先调用 `cc-run-batch`**
- **不通过再继续让 Claude Code 返工**

## 调度代理职责

调度代理在本项目中的职责只有：

1. 理解用户目标
2. 拆解成最小可执行任务
3. 调用 `cc-run` 让 Claude Code 执行
4. 读取执行结果、diff、报错
5. 进行验收
6. 若不达标，继续调用 `cc-run` 二次修改
7. 最终只把结果和高信号结论反馈给用户

对于批量重复任务，调度代理需要额外做到：

1. 先判断任务是否适合拆成多个独立批次
2. 适合拆分时，优先调用 `cc-run-batch`
3. 返工时优先局部返工，不整批重跑

## 进度文档

本项目的进度文档固定为：

- `Projects/00-当前进度.md`

每次项目发生修改，必须同步更新这份文档。

## 项目概述

这是一个基于 Obsidian 设计的个人知识管理系统，用于存储、整理和关联各类个人知识和信息。

## 项目结构

```text
PersonalKnowledgeBase/
├── Inbox/              # 收件箱 - 临时待整理内容
├── Daily/              # 每日笔记 - 日志和每日记录
├── Projects/           # 项目笔记 - 各类项目文档
├── Areas/              # 关注领域 - 长期关注的主题
├── Resources/          # 资源库 - 参考资料和链接
├── Archives/           # 归档 - 已完成/过时的内容
├── Templates/          # 模板 - 可复用笔记模板
├── .obsidian/          # Obsidian 配置
├── system-prompt.md    # AI 助手系统提示词
└── README.md           # 项目说明
```

## 核心约定

### 文件命名规范

- 文件名格式：`日期-主题描述.md` 或 `主题-描述.md`
- 示例：`2026-04-07-LangGPT提示词框架.md`
- 使用中文命名，保持简洁明了

### Markdown 格式规范

- 所有文档使用 Markdown 格式
- 必须包含 YAML frontmatter 元数据
- 使用标准标题层级（`#` → `##` → `###`）

### Obsidian 关联规范

- 使用双向链接：`[[笔记名称]]`
- 使用标签和属性，但 frontmatter 必须是标准 YAML
- 主动建立知识节点关联

### 分类规则

| 内容类型 | 存储位置 |
|----------|----------|
| 临时/待整理 | Inbox/ |
| 每日记录 | Daily/ |
| 项目相关 | Projects/ |
| 长期主题 | Areas/ |
| 参考资源 | Resources/ |
| 已完成/归档 | Archives/ |

## 本项目额外要求

- 不删除原始内容信息
- 不破坏现有双向链接关系
- 如果整理层和 `raw/` 冲突，以 `raw/` 为准，再由用户补充修正
- 交易复盘相关任务，默认优先维护：
  - `Projects/01-交易记录/`
  - `Projects/02-复盘总结/`
  - `Areas/2026-04-07-投资赚钱类型体系.md`
  - `Areas/2026-04-08-交易体系核心框架.md`
  - `Areas/2026-04-08-交易行动清单.md`
  - `Areas/2026-04-08-未来90天交易执行计划.md`

## 验收重点

调度代理验收时，至少检查：

- 任务是否真的完成
- 是否改了无关文件
- frontmatter 是否是标准 YAML
- wiki links 是否保留
- 内容是否更适合复习、检索、执行
- `Projects/00-当前进度.md` 是否同步更新
- 并行批次之间是否存在重复、漏项或格式漂移
